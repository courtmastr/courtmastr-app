Stats Structure: Sport → Category

Two-level stats on the global player document, dynamically keyed

/players/{playerId} — stats field
stats: {
  tennis: {
    singles:  { wins, losses, gamesPlayed, tournamentsPlayed }
    doubles:  { wins, losses, gamesPlayed, tournamentsPlayed }
    mixed:    { wins, losses, gamesPlayed, tournamentsPlayed }
  }
  pickleball: {
    singles:  { wins, losses, gamesPlayed, tournamentsPlayed }
    doubles:  { wins, losses, gamesPlayed, tournamentsPlayed }
    mixed:    { wins, losses, gamesPlayed, tournamentsPlayed }
  }
  badminton: {
    singles:  { wins, losses, gamesPlayed, tournamentsPlayed }
    doubles:  { wins, losses, gamesPlayed, tournamentsPlayed }
  }
}
Dynamic map — sports are added automatically as the player registers in new sports. No schema change needed when a new sport is introduced.
Tournament doc — one new field
{
  name: "Summer Open",
  sport: "tennis"  ← NEW
  orgId: "...",
  ...existing fields
}
One sport per tournament. The Cloud Function reads tournament.sport to know which bucket to update.
Category doc — already has type
{
  name: "Men's Singles",
  type: "singles"  ← already exists
  gender: "men",
  ...
}
Category type (singles/doubles/mixed) already exists. No change needed here.
Cloud Function: how it updates stats
1. Match result saved
match_scores written
→ trigger fires
→
2. Function reads
tournament.sport
category.type
registration.playerId(s)
→
3. Atomic update
stats.tennis.singles.wins
increment(1) on player doc
→
4. Doubles/Mixed
Both partners updated
in one batch write
Firestore FieldValue.increment() is atomic — no race conditions even if two matches finish at the same time.
