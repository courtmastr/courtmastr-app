When should the Cloud Function fire?

Per-match vs end-of-tournament — cost and accuracy trade-offs

ORIGINAL APPROACH
Trigger on every match result
A 64-player single elimination bracket
= 63 matches = 63 function calls
A round robin with 8 players
= 28 matches = 28 function calls
Per tournament with 3 categories:
~150–300 function invocations
Cost estimate
10 tournaments/month × 200 matches = 2,000 invocations
100 tournaments = 20,000 invocations
Free tier: 2M/month — fine at low scale
But reads per invocation add up fast
RECOMMENDED
Trigger once when tournament completes
Same 64-player bracket
= 1 function call at the end
Reads all match results in one go,
computes wins/losses per player,
batch-writes all global player stats
Per tournament: exactly 1 invocation
regardless of bracket size
Cost estimate
10 tournaments/month = 10 invocations
100 tournaments = 100 invocations
Practically free forever
One batch write per tournament

⚠️ Why statsProcessed matters
Without this flag, if an organizer accidentally sets status to "completed" twice, the function would run twice and double-count every win/loss. The statsProcessed: boolean field on the tournament prevents that. The function checks this first — if already true, it exits immediately. One tournament = one stats update. Safe to retry.
One more new field on Tournament
{
  orgId: "...",          ← links to org
  sport: "tennis",       ← drives stat bucket
  statsProcessed: false  ← NEW: prevents double-counting
}
Set to true by the Cloud Function after stats are written. Organizer can never accidentally trigger it twice.

This saves ~99% of function invocations vs per-match. Does this work for you?
