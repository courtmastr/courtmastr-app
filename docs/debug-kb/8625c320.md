# KB Entry: build:log fails when compatibility computed is reintroduced but not consumed

## Context
- Command: `npm run build:log`
- Working directory: `/Users/ramc/Documents/Code/courtmaster-v2`
- Recent changes: refreshed public-facing tournament shells and restored public schedule compatibility flags for tests.

## Error Signature
```text
error TS6133: 'hasPublishedSchedule' is declared but its value is never read.
```

## Attempts

### Attempt 1
**Date**: 2026-03-13
**Change**: Reintroduced `hasPublishedSchedule` in `PublicScheduleView.vue` so existing tests and integration checks could read the same computed flag after the public-shell refactor.

**Result**: ❌ `npm run build:log` failed because the computed was only exposed for tests and was not referenced by runtime code, so TypeScript treated it as unused before surfacing the unrelated baseline build failure.

### Attempt 2
**Date**: 2026-03-13
**Change**: Updated `shouldShowUnpublishedScheduleAlert` to depend on `hasPublishedSchedule`, keeping the compatibility computed in active runtime use while preserving the intended alert suppression when live schedule data is already visible.

**Result**: ✅ The local TypeScript error disappeared; subsequent builds returned only the known unrelated baseline fingerprint `20d5d4c3`.

## Root Cause
The public schedule compatibility computed was restored for tests but not consumed anywhere in the runtime view, so `noUnusedLocals` failed the build.

## Fix (final)
```bash
# In src/features/public/views/PublicScheduleView.vue
# - keep the restored computed:
const hasPublishedSchedule = computed(() => publishedMatches.value.length > 0);

# - consume it in runtime logic:
const shouldShowUnpublishedScheduleAlert = computed(() => (
  hasDraftMatches.value && !hasPublishedSchedule.value && nowPlayingItems.value.length === 0
));

npm run test -- --run tests/unit/PublicScheduleView.test.ts tests/integration/public-views.integration.test.ts
npx vite build
npm run build:log
```
